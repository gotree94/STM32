# DWT(Data Watchpoint and Trace) Cycle Counter vs GetTick(HAL_GetTick)

> STM32F103(Cortex-M3 코어 기반)에서 DWT(Data Watchpoint and Trace) Cycle Counter와 <br>
> GetTick(HAL_GetTick) 함수는 모두 시간이나 시스템의 지연(Delay)을 측정할 때 사용되지만, <br>
> 그 작동 원리와 정밀도(Resolution)에서 엄청난 차이가 있습니다.

* 두 방식의 핵심 차이점과 그 의미를 정리해 드립니다.

---

## 1. DWT Cycle Counter vs GetTick 비교

| 항목 | DWT Cycle Counter (DWT->CYCCNT) | HAL_GetTick () | 
|:---------------:|:---------------:|:---------------:|
| 기준 단위 | CPU 클럭 주기 (Cycle) | 밀리초 (ms, $10^{-3}$초) | 
| 하드웨어 출처 | Cortex-M3 내장 디버그/추적 장치 (DWT) | SysTick 타이머 인터럽트 | 
| 정밀도 (해상도) | 극도로 높음 (72MHz 기준 약 13.8ns, 64MHz 기준 약 15.625ns) | 낮음 (기본 1ms) | 
| 오버헤드 | 없음 (하드웨어 레지스터 직접 읽기) | 매우 적으나 인터럽트 의존적 | 
| 카운터 오버플로우 | 72MHz 기준 약 59.6초(64MHz 기준 약 67.1초)마다 발생 | 기본 32비트 변수 기준 약 49.7일 | 
| 주요 용도 | 함수 실행 시간 측정, 마이크로초($\mu s$) 단위 딜레이 | 시스템 업타임 확인, 밀리초 단위 스케줄링 | 

<br><br>

## 2. 세부적인 작동 원리와 차이의 의미

⚙️ DWT Cycle Counter<br>
* DWT는 ARM Cortex-M3 코어 내부에 내장된 디버그 모듈의 일부입니다. 
* DWT->CYCCNT 레지스터는 CPU 클럭이 튈 때마다 1씩 증가합니다.

   * 시간적 의미:
     * STM32F103을 최대 클럭인 64MHz로 구동할 경우, 1 사이클은 약 15.625나노초(ns)입니다.
     * 즉, 단 몇 줄의 어셈블리 명령어가 실행되는 시간까지 정확하게 측정할 수 있습니다.
   * 독립성:
     * 인터럽트를 타지 않고 하드웨어적으로 알아서 값이 올라가므로,
     * 다른 인터럽트가 걸려도 정확한 "클럭 소모량"을 측정합니다.
<br><br>

⏱️ HAL_GetTick()<br>
* 일반적으로 HAL_GetTick()은 SysTick 타이머 인터럽트를 기반으로 작동합니다.
* SysTick 타이머가 1ms마다 인터럽트를 발생시키면, ISR(인터럽트 서비스 루틴)에서 글로벌 변수(uwTick)를 1씩 증가시키는 방식입니다.

   * 시간적 의미:
     * 1ms($1,000,000\text{ ns}$) 단위로만 값이 변합니다.
     * 따라서 1ms 미만의 정밀도가 필요한 작업
     * (예: 센서 데이터 통신 타이밍, 미세한 코드 최적화 성능 측정)에는 전혀 사용할 수 없습니다.
   * 인터럽트 의존성:
     * 만약 HAL_GetTick()보다 우선순위가 높은 인터럽트가 오랫동안 실행되거나,
     * 코드 내에서 인터럽트를 완전히 비활성화(__disable_irq())하면 SysTick 카운터가 누락되어
     * 실제 시간보다 느리게 가는 현상이 발생할 수 있습니다.

<br><br>

## 3. 코드 관점에서의 의미와 활용 예시

① 성능 분석 및 프로파일링 (DWT 활용)
  * 어떤 알고리즘이나 함수의 순수 실행 시간을 측정하고 싶을 때는 DWT가 압도적으로 유리합니다.

```C
// DWT 활성화 (최초 1회 필요)
CoreDebug->DEMCR |= CoreDebug_DEMCR_TRCENA_Msk;
DWT->CTRL |= DWT_CTRL_CYCCNTENA_Msk;

// 측정 시작
uint32_t start_cycle = DWT->CYCCNT;

// [측정 대상 코드]
do_something_heavy();

// 측정 종료
uint32_t end_cycle = DWT->CYCCNT;
uint32_t total_cycles = end_cycle - start_cycle;

// 64MHz 기준 마이크로초(us) 변환
float microseconds = (float)total_cycles / 64.0f; 
// 72MHz 기준 마이크로초(us) 변환
//float microseconds = (float)total_cycles / 72.0f; 
```

② 일반적인 타임아웃 및 스케줄링 (GetTick 활용)

* 정밀도가 낮아도 상관없고 오랫동안 유지되는 타이밍
* (예: 1초마다 LED 플래싱, 500ms 통신 타임아웃 체크)에는 오버플로우 걱정이 없는 GetTick이 적합합니다.

```C
uint32_t last_time = HAL_GetTick();

while(1) {
    if (HAL_GetTick() - last_time >= 500) { // 500ms 주기
        HAL_GPIO_TogglePin(GPIOC, GPIO_PIN_13);
        last_time = HAL_GetTick();
    }
}
```

## 4. 요약

* 값의 차이:
  * DWT->CYCCNT는 클럭 주기 단위로 매우 빠르게 증가하는 초정밀 데이터이며,
  * HAL_GetTick()은 1ms마다 느리게 증가하는 소프트웨어적 데이터입니다.
* 선택의 기준:
  * 마이크로초($\mu s$) 단위 이하의 미세 제어나 함수 성능 측정이 필요하다면 DWT를,
  * 밀리초($ms$) 단위의 전반적인 시스템 흐름 제어 및 타임아웃 처리에는 GetTick을 사용하는 것이 맞습니다.

---

1. DWT Cycle Counter 사용 시 장·단점
Cortex-M3 코어 내부의 하드웨어 클럭을 직접 읽는 방식입니다.

👍 장점
나노초(ns) 단위의 압도적인 정밀도

64MHz 구동 시 1 사이클은 약 15.6ns, 72MHz 구동 시 약 13.8ns입니다. 코드 몇 줄이 실행되는 시간이나 미세한 신호의 타이밍을 완벽하게 잡아낼 수 있습니다.

소프트웨어 오버헤드 '제로(0)'

타이머 인터럽트를 가동하거나 누적 변수를 업데이트하는 소프트웨어 연산이 전혀 없습니다. 레지스터(DWT->CYCCNT)를 읽는 즉시 현재 시간이 측정되므로 시스템 성능에 영향을 주지 않습니다.

인터럽트 차단 시스템에서도 안전함

다른 아주 긴 인터럽트 서비스 루틴(ISR)이 실행 중이거나, 크리티컬 섹션 보호를 위해 인터럽트를 일시적으로 전면 비활성화(__disable_irq())하더라도 하드웨어 카운터는 멈추지 않고 독자적으로 계속 증가합니다.

👎 단점
매우 짧은 오버플로우 주기

32비트 크기 한계로 인해 64MHz 기준 약 67.1초가 지나면 카운터가 0으로 초기화됩니다. 분 단위나 시간 단위의 장기적인 시간 측정에는 직접 오버플로우 보정 코드를 작성해야 하므로 번거롭습니다.

디버그 모드 의존성 (이식성 제한)

DWT는 코어의 '디버그(Debug) 모듈' 일부입니다. 아주 일부 저가형 Cortex-M 코어(예: 일부 Cortex-M0/M0+ 라인업)에는 DWT 아키텍처가 아예 생략되어 있어, 코드를 다른 MCU로 옮길 때 호환되지 않을 수 있습니다. (STM32F103의 Cortex-M3에는 기본 포함되어 있습니다.)

2. HAL_GetTick() 사용 시 장·단점
SysTick 타이머 인터럽트를 통해 1ms마다 갱신되는 글로벌 변수를 참조하는 방식입니다.

👍 장점
매우 긴 오버플로우 주기 (장시간 안정성)

32비트 변수가 1ms마다 증가하므로, 오버플로우가 발생하기까지 약 49.7일이 걸립니다. 일반적인 임베디드 시스템의 타임아웃, 주기적 작업(태스크 스케줄링) 관리에 오버플로우 걱정 없이 안전하게 쓸 수 있습니다.

높은 이식성과 표준화

STMicroelectronics의 모든 MCU 제품군(STM32 F0, F1, F4, G0, H7 등)의 HAL 라이브러리에서 완벽히 동일한 문법과 동작을 보장하므로 코드 재사용성이 매우 높습니다.

가독성과 구현의 편리함

별도의 레지스터 활성화 설정 없이 HAL_GetTick() 호출만으로 직관적으로 시간을 계산할 수 있으며, HAL_Delay() 같은 기본 내장 함수들과 완벽히 연동됩니다.

👎 단점
낮은 정밀도 (최대 1ms의 한계)

기본 해상도가 1ms(1,000μs)이기 때문에 마이크로초(μs) 단위의 정밀한 지연(Delay)을 만들거나 센서와의 고속 통신 타이밍을 맞추는 용도로는 절대 사용할 수 없습니다.

인터럽트 의존성 및 누락 위험

만약 HAL_GetTick()의 인터럽트 우선순위(Priority)보다 높은 인터럽트가 발생하여 1ms 이상 CPU를 점유하거나, 코드 내에서 인터럽트를 비활성화하면 SysTick 카운터가 늘어나지 않습니다. 이로 인해 실제 시간보다 소프트웨어 시간이 느려지는 현상이 발생할 수 있습니다.

💡 한눈에 보는 요약 가이드
DWT를 써야 하는 상황: * "이 정렬 알고리즘을 수행하는 데 몇 클럭이나 소모될까?" (코드 프로파일링)

"DHT11 센서나 초음파 센서 신호를 받기 위해 50μs 동안 대기해야 한다." (μs 단위 정밀 딜레이 구현)

HAL_GetTick()을 써야 하는 상황:

"버튼이 눌린 후 3초 동안 디바운싱 처리를 하겠다." (초 단위 타임아웃)

"센서 데이터를 500ms 주기마다 주기적으로 읽어서 디스플레이에 뿌려주겠다." (메인 루프 주기 제어)

DWT Cycle Counter(DWT->CYCCNT)를 읽는 작업은 printf처럼 시스템 속도를 갉아먹거나 병목 현상(시간적/성능적 문제)을 일으키지 않습니다. printf와 DWT가 하드웨어 내부에서 어떻게 다르게 처리되는지 비교해 보면 그 이유를 명확히 이해할 수 있습니다.1. printf가 느리고 문제가 되는 이유임베디드 시스템에서 printf가 치명적인 성능 저하를 일으키는 이유는 다음과 같습니다.소프트웨어 오버헤드: 문자열을 파싱하고 형식을 변환(포맷팅)하는 복잡한 연산이 CPU 내부에서 수행됩니다.I/O 병목 (UART 등): 일반적으로 printf는 속도가 느린 직렬 통신(UART)을 통해 데이터를 한 바이트씩 전송합니다. 전송이 완료될 때까지 CPU가 대기(Blocking)하거나, 인터럽트가 수없이 발생하여 메인 루프를 방해합니다.2. DWT Cycle Counter가 안전한 이유 (성능 저하가 없는 이유)반면, DWT(DWT->CYCCNT) 값을 가져오는 행위는 단순한 내부 레지스터 읽기 작업입니다.단 1 사이클 소모 (가장 빠른 명령):DWT->CYCCNT 값을 읽어오는 C 언어 코드는 컴파일되면 어셈블리 명령어로 LDR (Load Register) 단 한 줄로 변환됩니다. Cortex-M3 아키텍처에서 이 명령은 일반적으로 단 1~2 CPU 클럭 사이클만에 완료됩니다. 64MHz 클럭 기준, 약 15~30나노초(ns) 밖에 걸리지 않습니다.하드웨어 독립 구동:카운터 자체는 CPU 코어와 별개로 하드웨어 실리콘 레벨에서 알아서 째깍째깍 올라가고 있습니다. CPU가 이를 읽을 때 코어의 연산을 방해하거나 멈추게 하지 않습니다.따라서 코드가 실행되는 도중에 DWT 값을 수십, 수백 번 읽더라도 시스템 성능이나 타이밍에 미치는 영향은 사실상 '제로(0)'에 가깝습니다.3. 그렇다면 DWT를 많이 쓸 때 주의해야 할 점(단점)은 전혀 없을까?printf 같은 시간적 병목은 없지만, DWT를 빈번하게 혹은 잘못 사용할 때 발생할 수 있는 주의점 3가지가 있습니다.① 잦은 연산으로 인한 부동소수점(float) 오버헤드 (가장 흔한 실수)DWT 값을 읽는 것 자체는 1 사이클이지만, 읽어온 사이클 값을 마이크로초($\mu s$)나 밀리초($ms$) 단위로 변환하기 위해 나눗셈 연산이나 부동소수점(float) 연산을 많이 추가하면 이때 성능이 떨어질 수 있습니다.이유: STM32F103(Cortex-M3)은 하드웨어 부동소수점 연산 장치(FPU)가 없습니다. 따라서 float 나눗셈을 수행하면 소프트웨어적으로 이를 에뮬레이션하기 위해 수십~수백 클럭이 소모됩니다.해결책: 성능 측정을 할 때는 루프 안에서 즉시 시간 단위 변환을 하지 말고, 순수 사이클 값(uint32_t)만 배열 등에 기록(로깅)해 둔 뒤, 모든 측정이 끝난 메인 루프나 덤프 함수에서 한 번에 시간 단위로 변환해야 합니다.② 너무 짧은 오버포화(Overflow) 주기 처리의 한계앞서 말씀드렸듯 64MHz 기준 약 67.1초마다 카운터가 다시 0으로 돌아갑니다. 만약 67초가 넘어가는 긴 작업을 DWT로만 측정하려고 하면, 단순히 종료시간 - 시작시간을 했을 때 엉뚱한 값이 나옵니다.잦은 오버플로우를 감지하기 위해 소프트웨어적으로 계속 상위 비트를 확장하는 코드를 넣다 보면 결국 코드가 복잡해지고 약간의 소프트웨어 오버헤드가 생길 수 있습니다.③ 디버거 연결 상태에 따른 전력 소모 및 비활성화 위험DWT는 기본적으로 디버깅을 위한 장치입니다. MCU의 저전력 모드(Sleep, Stop 등)에 진입할 때 디버그 모듈이 켜져 있으면 전류 소모가 예상보다 클 수 있습니다. 또한, 디버거(ST-Link 등)를 아예 연결하지 않고 하드웨어만 단독으로 부팅할 때, 초기화 코드(CoreDebug->DEMCR |= CoreDebug_DEMCR_TRCENA_Msk;)를 제대로 호출해주지 않으면 카운터가 동작하지 않아 딜레이 함수 등이 무한 루프에 빠질 위험이 있습니다.💡 요약printf와의 차이: printf는 시스템을 수 ms 동안 멈추게 할 수 있는 무거운 작업이지만, DWT 읽기는 수 ns 만에 끝나는 초경량 작업입니다.결론: DWT 레지스터를 많이 읽는 것 자체는 속도나 성능에 아무런 문제가 되지 않으므로 안심하고 프로파일링에 사용하셔도 됩니다. 단, 읽어온 값을 가지고 그 자리에서 float 나눗셈 연산을 남발하는 것만 주의하시면 됩니다.
