## 4. 모듈별 상세 분석

### 4.1 motor.c — 모터 구동

**기능:** 4개 DC 모터의 H-Bridge PWM 제어. signed duty 방식.

- **핵심:** `MOTOR_MAX_DUTY = 3199` (20kHz PWM, ARR=3199)
- **구조:** 각 모터는 forward/rev 채널 쌍으로 구성
  - LF: TIM2_CH2 (FWD) / TIM1_CH3 (REV)
  - RF: TIM3_CH1 (FWD) / TIM3_CH2 (REV) 
  - LR: TIM1_CH2 (FWD) / TIM2_CH3 (REV)
  - RR: TIM4_CH3 (FWD) / TIM4_CH4 (REV)
- **동작:** duty > 0 → FWD, duty < 0 → REV, duty = 0 → 정지

### 4.2 encoder.c — 홀 센서 엔코더

**기능:** DIY 홀 센서 + 자석(1개/바퀴) 기반 RPM 측정. 기간(Period) 기반 방식.

- **설계:** 1펄스 = 1회전 (단일 자석)
- `ENCODER_DEBOUNCE_FLOOR_MS = 20ms`: 20ms 미만 간격 펄스는 노이즈로 간주
- `ENCODER_STALL_TIMEOUT_MS = 2000ms`: 2초 이상 펄스 미수신 → 정지
- **ISR:** `Encoder_OnPulse()` — EXTI 인터럽트에서 호출, 타임스탬프 기록
- **샘플링:** `Encoder_Sample()` — 10ms 주기로 `RPM = 60000 / period_ms` 계산
- **한계:** 모터 구동 시 전자기 간섭(EMI) 및 자기적 바운스로 인한 노이즈 문제 존재

### 4.3 pid.c — PID 제어기

**기능:** 범용 PID (P, I, D + Anti-windup).

- `PID_Update()`: `output = Kp*e + Ki*∫e·dt + Kd*(de/dt)`
- `i_min` / `i_max`: 적분기 Anti-windup 클램프
- `out_min` / `out_max`: 출력 클램프
- **사용처:** 
  - `pid_speed[]` ×4: 속도 트림 (Kp=5, Ki=0, Kd=0, trim range ±800)
  - `pid_yaw`: 직진 요 보정 (Kp=0, Ki=0, Kd=0 — 미튠)
  - `pid_line`: 라인트레이싱 (Kp=0, Ki=0, Kd=0 — 미튠)

### 4.4 scurve.c — S-커브 속도 프로파일

**기능:** 저크(Jerk) 제한 S-커브 가감속.

- **상태:** ACCEL → CRUISE → DECEL → DONE
- `SCurve_Init()`: 목표 속도, 최대 가속도, 저크 설정
- `SCurve_Update()`: 가속도를 저크로 제한하며 증감, 목표 도달 시 CRUISE/STOP
- **사용처:** 직진(STRAIGHT) 가속, 장애물 회피(AVOID) 감속

### 4.5 kalman.c — 1차원 칼만 필터

**기능:** 자이로 각속도와 가속도 각도를 융합하여 요(Yaw) 추정.

- **상태 변수:** `angle` (요 각도), `bias` (자이로 바이어스)
- **공분산 행렬:** 2×2 (`covariance[2][2]`)
- **Q:** `q_angle=0.001`, `q_bias=0.003`
- **R:** 실시간 측정 (`Kalman_SetR()`: 정지 중 분산 측정)
- **플래시 저장:** Kalman R과 bias를 Flash 메모리에 저장/로드 가능

### 4.6 mpu6050.c — MPU-6050 IMU 드라이버

**기능:** 소프트웨어 I2C를 통한 MPU-6050 센서 데이터 읽기.

- **초기화:** WHO_AM_I 확인(0x68) → PWR_MGMT_1 → GYRO_CONFIG(±250°/s) → ACCEL_CONFIG(±2g) → CONFIG(DLPF)
- **읽기:** ACCEL_XOUT_H~GYRO_Z까지 14바이트 읽기
  - `omega_dps = gyro_z / 131.0` (FS_SEL=0)
  - `accel_angle = atan2(accel_y, accel_z) × 57.2958` (중력 기준 틸트)

### 4.7 soft_i2c.c — 소프트웨어 I2C

**기능:** 비트뱅 I2C 마스터 (SCL=PA6, SDA=PB11).

- **타이밍:** `SystemCoreClock / 800000` ≈ 80kHz I2C
- **지원:** Standard I2C 프로토콜 (START, STOP, ACK/NACK, byte write/read)
- **API:** MemRead, MemWrite, MemRead16, MemWrite16, MasterTransmit, IsDeviceReady

> **배경:** 하드웨어 I2C1을 사용하면 PB5에 SMBA 신호가 나타나 TIM3_CH2(RF-REV)를 무력화시키는 문제 발견. 모든 하드웨어 I2C 배치가 모터 PWM과 충돌하여 소프트웨어 I2C 채택.

### 4.8 line_sensor.c — 라인트레이서

**기능:** ADC1 + DMA로 라인 센서 읽기.

- **DMA:** ADC1 CH0, 8개 샘플 circular buffer
- **콜백:** `HAL_ADC_ConvHalfCpltCallback` / `ConvCpltCallback`에서 fresh 플래그 설정
- **처리:** DMA 버퍼 평균 → 정규화 → 에러 값 (center 기준 편차)
- **설정:** `LineSensorConfig`로 raw min/max/center/invert 튜닝 가능

### 4.9 hc06.c — HC-06 블루투스 명령 처리

**기능:** USART2 DMA RX로 AT 명령 수신 및 파싱.

- **명령 테이블:**
  | 명령 | 동작 |
  |---|---|
  | `AT+FWD` | 직진(STRAIGHT) 모드 |
  | `AT+LEFT` / `AT+RIGHT` | 90° 선회(TURN) |
  | `AT+STOP` | 정지(IDLE) |
  | `AT+MAN` | 수동 모드(MANUAL) |
  | `AT+LINE` | 라인트레이싱(LINE_TRACE) |
  | `AT+RST` | 비상 복구(EMERGENCY → IDLE) |
  | `AT+GET` | 파라미터 조회(R, BIAS, YAW) |
  | `AT+SAVE` | 파라미터 플래시 저장 |

- **DMA RX:** 1바이트씩 DMA 수신, `\r`/`\n`에서 라인 완료. 명령 버퍼는 ISR → 메인루프에서 처리.

### 4.10 lcd.c — I2C LCD (PCF8574)

**기능:** 소프트웨어 I2C를 통한 16×2 캐릭터 LCD 제어.

- **4비트 모드:** I2C 백팩(PCF8574)을 통한 HD44780 호환
- **표시:** 1라인: 상태명, 2라인: Yaw 각도
- **주소:** 0x27 (7비트)

### 4.11 vl53l1x.c — VL53L1X ToF 거리 센서

**기능:** 3개의 VL53L1X ToF 센서 (전방/좌측/우측) 장애물 감지.

- **XSHUT 순서:** 전부 LOW → Front 활성화(주소 0x54) → Left 활성화(0x56) → Right 활성화(0x58)
- **설정:** Short Distance Mode, Timing Budget 50ms, Inter-Measurement 70ms
- **임계값:** 전방 250mm, 측면 180mm 이하 → 장애물
- **ST API:** STSW-IMG007 공식 API 사용 (Drivers/VL53L1X/)

### 4.12 paramstore.c — 파라미터 플래시 저장

**기능:** 칼만 필터 R값과 bias를 Flash 메모리에 저장/로드.

- **주소:** `0x0801FC00` (마지막 페이지, 크기 1KB)
- **Magic:** `0x4B414C31` (유효성 검증)
- **저장:** Flash 페이지 소거 → Word 단위 Program (magic, R, bias)

### 4.13 fsm.c — 상태 머신

**기능:** 7개 상태를 가진 중앙 제어 FSM.

#### 상태 전이도

```
                   ┌─────────────┐
                   │   IDLE      │◄──── (초기 상태)
                   └──────┬──────┘
                          │
            ┌─────────────┼──────────────┐
            │             │              │
       AT+FWD        AT+LINE         AT+MAN
            │             │              │
            ▼             ▼              ▼
     ┌──────────┐  ┌────────────┐  ┌─────────┐
     │ STRAIGHT │  │ LINE_TRACE │  │ MANUAL  │
     └──┬───┬───┘  └──────┬─────┘  └─────────┘
        │   │             │               │
   AT+LEFT│  │       장애물               │ AT+STOP
   AT+RIGH │  │         감지              │
        │   │             │               │
        ▼   │             ▼               │
   ┌───────┐│        ┌────────┐           │
   │ TURN  ││        │ AVOID  │           │
   └───┬───┘│        └───┬────┘           │
       │    │            │                │
       │    └────┬───────┘                │
       │         │ 회피 완료              │
       │         ▼                        │
       └─────→ STRAIGHT ◄─────────────────┘
                         │
                    비상(EMERGENCY)
                         │
                    AT+RST → IDLE
```

#### 상태 설명

| 상태 | 동작 | 전환 조건 |
|---|---|---|
| **IDLE** | 모든 모터 정지 | 초기 진입 |
| **STRAIGHT** | 요-PID 보정 직진, S-커브 가속 | AT+FWD, TURN 완료, AVOID 완료 |
| **LINE_TRACE** | 라인-PID 기반 주행 | AT+LINE |
| **TURN** | 자이로 적분 기반 90° 선회 | AT+LEFT/RIGHT (STRAIGHT에서) |
| **AVOID** | S-커브 감속 → 좌선회 → 전진 → 복귀 | 장애물 감지 |
| **MANUAL** | 수동 방향 제어 | AT+MAN |
| **EMERGENCY** | 즉시 정지, 모든 모터 OFF | 비상 요청 (ISR) |

#### AVOID 상세 시퀀스

```
AVOID 진입 → [DECEL] S-커브 감속(80→0 RPM) → [TURN] 90° 좌회전
→ [CLEAR] 300ms 전진 → STRAIGHT 복귀
```

## 5. 메인 루프 시간 흐름

### 5.1 초기화 순서

```
HAL_Init() → SystemClock_Config()
→ MX_GPIO_Init() → MX_DMA_Init() → MX_ADC1_Init() → MX_USART2_UART_Init()
→ MX_TIM1_Init() → MX_TIM2_Init() → MX_TIM3_Init() → MX_TIM4_Init()
→ Motor_Init() → Encoder_Init() → FSM_Init() → HC06_Init()
→ SoftI2C_Init() → TOF_Init_All() → LineSensor_Init() → MPU6050_Init()
→ Kalman_Init() → ParamStore_Load() → speed_PID_Init() → LCD_Init()
→ MotorTest_Run() or MotorSweep_Run() (조건부)
```

### 5.2 메인 루프 (슈퍼루프)

```
while (1):
  ├── HC06_Process()                     # 매 루프: 블루투스 명령 처리
  ├── ParamStore 처리 (조건부)
  ├── dt 계산 (HAL_GetTick)
  │
  ├── [10ms 주기] Speed Loop:
  │     Encoder_Sample(dt)
  │     for each wheel:
  │       setpoint = FSM_GetSpeedSetpoint()
  │       measured = Encoder_GetSpeed()
  │       trim = PID_Update(pid_speed, setpoint, measured)
  │       duty = SpeedFF_Duty(setpoint) + trim
  │       Motor_SetDuty(duty)
  │
  ├── MPU6050_Read() → Kalman_Update()   # 매 루프 (가변)
  │     + 정지 중 R 측정 (200샘플 윈도우)
  │
  ├── LineSensor_Read()                  # 매 루프
  │
  ├── [50ms 주기] TOF_UpdateAll()        # VL53L1X 거리 갱신
  │
  ├── FSM_Dispatch(yaw, omega, dt, ...)   # 매 루프
  │
  ├── [50ms 주기] UART Stream 출력       # YAW, SPD, CNT, TOF, OBS 텔레메트리
  │
  └── [200ms 주기] LCD_Update()          # LCD 상태 표시
```

### 5.3 인터럽트 서비스 루틴

| ISR | 트리거 | 동작 | 우선순위 |
|---|---|---|---|
| EXTI0_IRQHandler | PB0 Rising | `Encoder_OnPulse(MOTOR_LF)` | 1 |
| EXTI1_IRQHandler | PB1 Rising | `Encoder_OnPulse(MOTOR_RF)` | 1 |
| EXTI4_IRQHandler | PA4 Rising | `Encoder_OnPulse(MOTOR_RR)` | 1 |
| EXTI9_5_IRQHandler | PB6 Rising | `Encoder_OnPulse(MOTOR_LR)` | 1 |
| DMA1_Ch1_IRQHandler | ADC1 DMA 완료 | HAL DMA 핸들러 → ADC 콜백 | 0 |
| DMA1_Ch6_IRQHandler | USART2 RX | HAL DMA → HC06_OnReceive | 2 |

## 6. 주요 함수 기능 설명

### 모터 제어
| 함수 | 설명 |
|---|---|
| `Motor_Init()` | 모든 PWM 채널 시작, 듀티 0 초기화 |
| `Motor_SetDuty(id, duty)` | signed duty 설정 (±3199). 양수=FWD, 음수=REV |
| `Motor_SetDirection(id, dir)` | 전진/후진/정지 편의 함수 |
| `Motor_StopAll()` | 모든 모터 정지 |

### 엔코더
| 함수 | 설명 |
|---|---|
| `Encoder_Init()` | 카운트/속도/주기 변수 초기화 |
| `Encoder_OnPulse(wheel)` | ISR에서 호출. 펄스 도착 시간 기록, 주기 갱신 (디바운스 적용) |
| `Encoder_Sample(dt)` | 최신 주기 → RPM 변환 (`60000/period_ms`). 2초 정체 시 0 |
| `Encoder_GetSpeed(wheel)` | 현재 RPM 반환 |

## 7. 잘된 점 (Strengths)

### 7.1 아키텍처 설계
1. **명확한 모듈 분리:** 각 기능이 별도 .c/.h 파일로 분리되어 유지보수 용이
2. **상태 머신 기반 제어:** 7개 상태로 모든 동작을 명확히 정의하고 전이 규칙 체계화
3. **피드포워드 + PID 구조:** Feedforward Duty로 대부분의 구동력을 제공하고 PID는 미세 트림만 담당하여 응답성과 정밀도 균형

### 7.2 빌드 시스템
1. **CMake 기반:** CubeIDE 의존성 없음, CI/CD 및 CLI 빌드에 적합
2. **CubeMX + 수동 코드 혼용:** `.ioc`로 하드웨어 설정, `USER CODE` 블록에 애플리케이션 로직 분리 성공

### 7.3 하드웨어 문제 해결
1. **I2C SMBA 문제 발견 및 우회:** PB5에 SMBA 신호가 PWM을 무력화시키는 근본 원인을 찾아 소프트웨어 I2C로 전환
2. **핀 충돌 해결:** Shock 센서 제거, HALL_RL 핀 이동(PB7→PB6), SCL 이동(PB6→PA6) 등 실제 배선에 맞춰 지속적 조정

### 7.4 품질 관리
1. **단위 테스트 존재:** `tests/test_encoder.c`, `tests/test_pid.c` — 호스트 환경에서 알고리즘 검증
2. **Python 스크립트:** 속도 데이터 수집 및 피드포워드 계수 피팅 자동화
3. **세부 문서화:** plan/ 디렉토리에 OVERVIEW, ARCHITECTURE, PHASES, REVIEW, USER, DECISIONS 등 체계적 문서화

### 7.5 코드 품질
1. **동적 할당 없음:** 제한된 SRAM(20KB)에서 정적 할당만 사용
2. **ISR 안전성:** `__get_PRIMASK()`/`__disable_irq()`로 공유 데이터 보호
3. **디바운스 및 타임아웃:** 엔코더 노이즈 대비(20ms), 정체 감지(2s)
4. **안전장치:** 비상 상황(EMERGENCY)에서 즉시 모터 정지

## 8. 개선 필요 사항 (Weaknesses & Improvements)

### 8.1 하드웨어 문제 (Critical)
1. **엔코더 노이즈 (D18/D20):** 모터 구동 시 홀 센서 라인에 EMI/자기 바운스로 인한 노이즈 발생. RPM이 10,000 이상으로 튀는 현상.
   - **대책:** `ENCODER_DEBOUNCE_FLOOR_MS` 튜닝, 하드웨어 RC 필터, 슈미트 트리거 홀 센서, 자석-센서 간극 조정 필요
2. **ToF 장애물 미감지 (D19):** 실제 충돌 상황에서도 OBS=0 유지.
   - **대책:** 센서 장착 각도/높이 재검토, 임계값 조정, Range Status 분석

### 8.2 미튜닝 파라미터
1. **모든 PID 게인이 0:** `pid_yaw`(Kp=0), `pid_line`(Kp=0) — 실제 주행 튜닝 전무
2. **회전 각도 목표:** `FSM_TURN_TARGET_DEG = 90°` 고정값 — 실제 테스트 필요
3. **가속/저크 한계:** `FSM_ACCEL_RATE_RPM_S`, `FSM_JERK_RPM_S2` — 하드웨어 검증 필요

### 8.3 구조적 개선
1. **슈퍼루프 한계:** 모든 처리가 메인루프에서 순차 실행. 시간 민감 작업(특히 속도 제어)의 지연 가능성
   - **제안:** TIM 기반 Systick 대신 PWM TIM의 Update 이벤트로 속도 제어 인터럽트 구현 고려
2. **매직 넘버:** 여러 상수가 #define 없이 코드 내 하드코딩
3. **에러 처리 부족:** `Error_Handler()`가 단순 무한루프 — 디버깅 어려움
   - **제안:** 에러 코드를 GPIO/UART로 출력하거나 LCD에 표시

### 8.4 기능적 개선
1. **속도 PID 적분 미사용:** `SPEED_KI=0` — 정상상태 오차 존재. 노이즈 문제 해결 후 Ki 추가 필요
2. **UART 과부하:** 50ms마다 YAW + SPD + CNT + TOF + OBS 동시 출력 — 저속 블루투스에서 버퍼 오버플로우 위험
3. **바퀴별 PID 공유 안됨:** 각 바퀴 독립 PID지만, 네 바퀴 사이의 협력 제어 부재 (예: 차체 기울기 고려)

### 8.5 빌드/테스트
1. **호스트 테스트만 존재:** 단위 테스트가 호스트에서만 실행, 타겟에서의 자동화된 통합 테스트 부재
2. **디버그 UART 공유:** HC-06와 벤치 디버그가 USART2 공유 — 동시 사용 불가

### 8.6 코드 스타일
1. **혼재된 코딩 컨벤션:** 함수 이름에 CamelCase(`FSM_Dispatch`)와 SnakeCase(`Encoder_OnPulse`) 혼용
2. **주석 부족:** 복잡한 알고리즘(칼만, S-커브)에 설명 주석 부족
3. **들여쓰기 불일치:** 일부 파일에서 2칸/4칸 스페이스 혼용

## 9. AI 협업 분석 (AI-Assisted Development)

### 9.1 plan/ 디렉토리 구조 및 역할

`plan/` 디렉토리는 AI(Claude)와 협업하여 프로젝트를 진행하기 위한 완전한 설계 문서 세트:

| 파일 | 역할 |
|---|---|
| `README.md` | 현재 상태, 차기 작업, 문서 맵 |
| `OVERVIEW.md` | 목표, 플랫폼, 범위, 제약사항 |
| `ARCHITECTURE.md` | 디렉토리 구조, 콜 그래프, 모듈 계약 |
| `PHASES.md` | 단계별 개발 워크플로우 (하드웨어 게이트 → CubeMX → 코드 → 검증) |
| `DECISIONS.md` | 결정 로그 (20개 항목), 선택 이유, 근거 |
| `USER.md` | 핀맵, 배선도, 빌드 명령어 |
| `REVIEW.md` | 허용 기준, 단계별 검증 상태, 데이터 수집 절차 |

### 9.2 AI 활용 방식

이 프로젝트는 AI(Claude)가 **설계, 코드 생성, 문서화, 디버깅, 하드웨어 문제 해결** 전반에 깊이 관여한 사례:

1. **설계 단계:** 사용자의 `plan.md`를 기반으로 아키텍처 설계 및 벤치마킹 (유사 오픈소스 프로젝트 참조)
2. **CubeMX 자동화:** STM32CubeMX를 헤드리스 CLI로 실행하여 `.ioc` → CMake 생성 자동화 (D5)
3. **단계별 개발 (Phased Approach):** 하드웨어 의존성에 따른 9단계(Phase 0~9) 개발
4. **지속적 문서화:** 모든 결정사항(DECISIONS.md)을 시간순으로 기록, 문제 해결 과정 체계화
5. **RAG 시스템 (.plan-rag/):** Chroma 벡터 DB + SQLite 기반 RAG 저장소로, AI 세션 간 컨텍스트 유지
   - `chroma/chroma.sqlite3`: 임베딩 벡터 저장소
   - `proposals/*.json`: 제안 변경사항 (evidence + changes + hash)
   - `plan.db`: 문서 메타데이터 DB
   - `graph.html`: 문서 관계 시각화

### 9.3 RAG 저장소 구조 (.plan-rag/)

```
.plan-rag/
├── chroma/
│   └── chroma.sqlite3    # 벡터 임베딩 DB
├── proposals/
│   ├── rfsbp...json      # 제안 1: HALL_RL PA6 할당, 문서 업데이트
│   └── xVFzZ...json      # 제안 2: 엔코더 재설계 (period-based)
├── graph.html            # 문서 연결 그래프 (시각화)
├── plan.db               # plan 문서 메타데이터
├── plan.db-shm
└── plan.db-wal
```

각 proposal은:
- `evidence[]`: 변경 근거 (실제 코드 탐색 결과, 사용자 확인)
- `changes[]`: 변경 대상 파일, 변경 유형, 원본 해시 포함
- 버전 관리 + 변경 내역 추적 가능

### 9.4 AI 협업 평가

**잘된 점:**
1. **결정 로그 체계화:** D1~D20까지 20개의 결정을 시간순 기록. 문제 발생 시 원인 분석과 해결 과정을 구조화
   - 예: D10(I2C SMBA 문제 → Soft I2C), D16(Shock 제거 → HALL_RL 이동 → SCL 이동)
2. **증거 기반 개발:** 단순 추측이 아닌 코드 탐색 결과를 근거로 제시 (각 proposal의 evidence[])
3. **단계적 접근:** Phase 0~9로 나누어 하드웨어 의존성에 따른 체계적 개발
4. **문제 격리 능력:** D18(EMI 노이즈) → D20(자기적 바운스 재현)으로 원인 정정하는 과정은 AI의 분석 능력 우수

**개선 필요 사항:**
1. **문서와 코드의 이중 관리:** plan/ 문서를 최신으로 유지하는 오버헤드. 코드 변경마다 문서 업데이트 필요
2. **제안 검증 부재:** proposals의 변경사항이 실제로 적용되었는지 확인하는 메커니즘 부족
3. **하드웨어 한계:** AI는 하드웨어 문제(EMI, 센서 장착 위치)에 대한 진단은 가능하나, 물리적 해결책(필터링, 구조 변경)은 제안에 그침
4. **자동화 한계:** CubeMX 헤드리스 생성은 가능하나, 핀 배치/주변장치 설정은 수동 `.ioc` 편집 필요
