# 4조 LCD-SPI (MicroGuard AMR) 프로젝트 분석 리포트

## 1. 프로젝트 개요

| 항목 | 내용 |
|---|---|
| **MCU** | STM32F103RBT6 (NUCLEO-F103RB, LQFP64) |
| **Clock** | 64 MHz (HSI 8MHz ÷2 → PLL x16) |
| **IDE** | STM32CubeIDE v6.14.1, HAL FW_F1 V1.8.7 |
| **.ioc** | `LCD-SPI.ioc` |
| **프로젝트명** | **MicroGuard AMR Lite** — 자율주행 모바일 로봇 플랫폼 |
| **코드 규모** | 22개 .h + 23개 .c, 약 4500줄 사용자 코드 |

### 프로젝트 구조 (모듈 계층)

```
main.c / main.h  ← CubeMX 초기화 + ST7735 SPI LCD 드라이버 내장

      │
      ├── app_config.h         ← 전역 설정 상수 (매직 넘버 제거)
      ├── app_types.h          ← 공용 타입 (AppMove_t, AppContext_t)
      ├── app_hw.c/h           ← HW 추상화 계층 (micros, delayUs, ADC, GPIO)
      ├── app_safety.c/h       ← 안전 계층 (Motor Arm, Emergency Latch, IWDG)
      ├── app_motor.c/h        ← 4륜 독립 PWM 모터 (TIM1~4, 리맵, 램프, 부스트)
      ├── app_core_sensing.c/h ← 핵심 센서 (초음파 EXTI 비동기, 라인, IR, 범퍼, CDS)

      ├── app_sensor_authority.c/h ← 센서 권한 집계 (정지/알람/조이스틱)
      ├── app_protocol.c/h     ← 직렬 프로토콜 (IRQ 링 버퍼, JSON ACK/State)
      │
      ├── app_buzzer.c/h       ← 부저 (PB7, 비동기, 경찰 사이렌)
      ├── app_clcd_ui.c/h      ← CLCD 디스플레이
      ├── app_tft_ui.c/h       ← TFT ST7735 UI 계층
      ├── app_status_indicator.c/h ← NeoPixel (PA3, UART 비트뱅)

      └── payload_*.c/h        ← 페이로드 센서 모듈

           └── payload_diagnostic_deck.c/h← 기울기/충격 센서
```

## 3. 모듈별 상세 분석

### 3.1 app.c — MicroGuard Runtime (App_Init / App_Service)

**App_Init()** — 부트 시퀀스 (단계별 진행률 출력):
1. AppHw → Safety → Protocol → Buzzer → CLCD → StatusIndicator → BootVisual
2. CoreSensing → LineTrace (24%)
3. SensorAuthority (40%)
4. Environment → OperatorInputs → MPU6050 → DiagnosticDeck (62%)
5. Motor (82%) → READY (100%)
6. BootVisual_Complete → TFT 보라색 대시보드 → 부저 1.8kHz → 부트 JSON 전송

**App_Service()** — 메인 루프 (각 모듈의 Service 호출):
```
1. Protocol_Service()      ← UART RX 처리 (USB + BT IRQ 링 버퍼)
2. CoreSensing_Service()   ← 센서 샘플링 (120ms 주기)
3. LineTrace_Service()     ← 라인트레이싱 자동 주행 (120ms)
4. Environment_Service()   ← DHT11/화염/사운드 (250ms)
5. OperatorInputs_Service() ← 조이스틱/터치/IR 리모컨 (100ms)
6. App_HandleRemoteMove()  ← IR 리모컨 이동 명령 처리
7. MotionMpu6050_Service() ← MPU6050 (500ms)
8. DiagnosticDeck_Service() ← 기울기/충격 (100ms)
9. SensorAuthority_Service() ← 센서 권한 집계 → 정지/알람/조이스틱
10. Motor_Service()         ← 모터 램프 + PWM 출력 (10ms)
11. Buzzer_NotifyMove() / Buzzer_Service()
12. ClcdUi_Service()        ← CLCD 갱신 (1000ms)
13. StatusIndicator_Service() ← NeoPixel (1000ms)
14. TftUi_Service()         ← TFT 갱신
15. Safety_WatchdogRefresh() ← IWDG 리프레시 (500ms)
16. 1초마다 Protocol_SendStatus() ← 전체 JSON 상태 전송
```

### 3.2 app_safety.c — 안전 계층

| 함수 | 기능 |
|------|------|
| `Safety_Init` | 컨텍스트 초기화 (motor_armed=0, emergency_latched=0) |
| `Safety_CanDrive` | emergency_latched / motor_armed 검사 |
| `Safety_SetMotorArmed` | M 명령 → arm / U 명령 → disarm |
| `Safety_StopNow` | X/0 명령 → 즉시 Motor_StopNow |
| `Safety_LatchEmergency` | B1/범퍼/화염/기울기/MPU 기울기 → 복구 필요 정지 |
| `Safety_WatchdogInit` | IWDG 레지스터 설정 (PR=6, RLR=2047) |
| `Safety_WatchdogRefresh` | App_Service() 마다 500ms 주기로 refresh |

**Watchdog 동작**: fault loop에 진입하면 refresh가 중단되어 약 1.6초 후 IWDG 리셋 발생. 리셋 전에 `Motor_EmergencyCutoffFromFault()`가 모든 TIM 출력을 레지스터 레벨에서 즉시 차단.

### 3.3 app_motor.c — 4륜 독립 PWM 모터

**핵심 설계**:
- TIM1~4 사용 (TIM1: LF, TIM2: RF, TIM3: LB, TIM4: RB)
- 각 휠마다 정/역 2채널 → 8개 PWM 채널
- PB8/PB9 I2C1 충돌 해결: 모터 Init 시 I2C1 PE 비활성화 + 클럭 차단

**모터 매트릭스** (2-side skid steering):

| 명령 | Left | Right | 설명 |
|------|------|-------|------|
| W | +Base | +Base | 전진 |
| S | -Base | -Base | 후진 |
| Q | +Curve | +Base | 좌회전 전진 |
| E | +Base | +Curve | 우회전 전진 |
| A | -Point | +Point | 좌축회전 |
| D | +Point | -Point | 우축회전 |
| Z | -Curve | -Base | 좌회전 후진 |
| C | -Base | -Curve | 우회전 후진 |

**램프 & 부스트 시스템**:
- `Motor_MoveToward()`: actual → target 방향으로 step=18씩 이동 (저역 통과)
- `Motor_SetSideTarget()`: 출발 시 110ms static boost (850 duty)
- `Motor_UpdateSide()`: 방향 반전 시 zero-cross dwell (40ms 정지 후 재출발)
- `Motor_OutputSide()`: boost 유효 시간 동안 최소 출력 유지

### 3.4 app_core_sensing.c — 핵심 센서

**초음파 — EXTI 비동기 상태머신**:
```
CORE_US_IDLE
  → (TRIG 발사) → CORE_US_WAIT_RISE
  → (EXTI Rising) → CORE_US_WAIT_FALL  
  → (EXTI Falling) → pulse_us 계산 → CORE_US_IDLE
```
- 타임아웃: RISE 3.5ms, FALL 25ms
- 거리: `pulse_us / 58` (cm)
- 두 센서 번갈아 가며 측정 (`s_next_ultrasonic` 토글)
- `CoreSensing_ServiceUltrasonic()`로 타임아웃 감시

**라인 센서**:
- PC8(Left), PC9(Right), Active LOW (검은색=0)
- `CoreSensing_LineState()`: DUAL_RAIL / LEFT_WHITE / RIGHT_WHITE / LOST 판정

**기타**: CDS(ADC), IR 장애물(PB13, Active LOW), 범퍼(PB15, Active LOW)

### 3.5 app_line_trace.c — 자동 라인트레이싱

| 상태 | 센서 (L,R) | 모터 명령 |
|------|------------|----------|
| BLACK / BLACK | 직진 | W (전진) |
| WHITE / BLACK | 좌보정 | Q (좌회전 전진) |
| BLACK / WHITE | 우보정 | E (우회전 전진) |
| WHITE / WHITE | 정지 | LOST_STOP |

- 수동 조작 시 `LineTrace_Suppress()`로 1200ms 동안 자동 보정 억제
- `Safety_CanDrive()`를 통과해야만 모터 명령 전달

### 3.6 app_sensor_authority.c — 센서 권한 집계

**정지 조건 (Stop Authority, 우선순위)**:
1. B1 버튼 → Latch
2. 범퍼 → Latch
3. IR 장애물 → 일반 정지
4. 초음파 ≤ 18cm → 일반 정지
5. 화염 감지 → Latch
6. 기울기 센서 → Latch
7. 충격 센서 → Latch
8. MPU6050 기울기 ≥ 12000 → Latch
9. 조이스틱 SW → Latch

**알람 조건**: 사운드 ≥ 3200 ADC / 터치 감지 → 경찰 사이렌 모드

**조이스틱 주행**: X/Y ADC를 8방향 매트릭스로 매핑 (Joystick Drive)

### 3.7 app_protocol.c — 직렬 프로토콜

**링 버퍼** (IRQ-safe):
- USB RX: 64바이트 링 (USART2, IRQ)
- BT RX: 64바이트 링 (USART3, IRQ)
- `Protocol_UartIrq()`: ORE/NE/FE/PE 오류 카운트 → RingPush

**한 글자 명령 체계**:

| 명령 | 기능 |
|------|------|
| M | Motor Arm (주행 준비) |
| U | Motor Disarm |
| R | Emergency Clear |
| X / 0 | 즉시 정지 |
| Q/W/E/A/S/D/Z/C | 수동 주행 (8방향) |
| 1~9 (IR Remote) | 리모컨 주행 (동일 매트릭스) |
| T | 라인트레이싱 ON |
| N | 라인트레이싱 OFF |
| F + 1~6 | 바퀴 Proof 테스트 |
| P | TFT 보라색 대시보드 |
| B | 부저 비프 |
| V | 검증 JSON |
| L / ? | 전체 상태 JSON |

**JSON 상태 프로토콜**: 규모 약 5KB — 모든 모듈 상태를 집계하여 USB/BT 동시 전송

### 3.8 main.c — ST7735 TFT LCD 드라이버

- ST7735S 0.96" 160x80 (RGB565), SPI 1Line 16MBit/s
- 소프트웨어 리셋 → Sleep Out → 프레임레이트/전원/감마 설정 → Display On
- 8x8 폰트 (ASCII 32~126) 내장
- `LCD_DrawChar()`: LSB 우선 비트맵 렌더링
- `LCD_Fill()`: 160x80개 픽셀 순차 SPI 전송 (blocking)

## 4. 시스템 타이밍 흐름

```
[Power On]
  ├─ HAL / Clock Init (HSI → PLL 64MHz)
  ├─ MX_GPIO / USART2 / I2C1 / SPI1 Init
  ├─ LCD_Init() (ST7735, ~850ms delay)
  ├─ App_Init() (5단계 부트, 각 70ms, 총 ~500ms)
  │   └─ 모터 Init (TIM 리맵, 8채널 PWM 시작, I2C1 비활성화)
  │   └─ 센서 Init (EXTI0/1, GPIO, ADC)
  │   └─ Boot Visual 완료 → TFT 대시보드
  └─ App_Service() 무한 루프
       │
       ┌─── Service Tick (비동기, 각 모듈 자체 주기) ───┐
       │                                                 │
       ├─ Protocol_Service()     ← 매 루프 (링버퍼 소비)
       ├─ CoreSensing_Service()  ← 120ms (센서 샘플링)
       │   └── EXTI0/1 (비동기 초음파 에지)
       ├─ LineTrace_Service()    ← 120ms (자동 주행)
       ├─ Environment_Service()  ← 250ms
       ├─ OperatorInputs_Service() ← 100ms
       ├─ App_HandleRemoteMove() ← 매 루프 (IR 명령)
       ├─ MotionMpu6050_Service() ← 500ms (I2C)
       ├─ DiagnosticDeck_Service() ← 100ms
       ├─ SensorAuthority_Service() ← 100ms (센서 집계)
       ├─ Motor_Service()        ← 10ms (ramp update)
       ├─ Buzzer_Service()       ← 매 루프 (비동기)
       ├─ ClcdUi_Service()       ← 1000ms
       ├─ StatusIndicator_Service() ← 1000ms
       ├─ TftUi_Service()        ← 매 루프 (변경 감지)
       └─ Safety_WatchdogRefresh() ← 500ms (IWDG)
```

**핵심 특징**: 각 모듈이 자체 주기(ms)를 가지고 last_service_ms 기준으로 동작. `App_Service()` 자체는 blocking 없이 매 루프 실행됨. 센서 authority가 즉시 정지를 내리면 모터 CCR이 레지스터 레벨에서 차단.

## 5. 주요 설계 결정 및 트레이드오프

### 5.1 PB8/PB9 I2C1 ↔ TIM4 충돌

- CubeMX에서는 PB8/PB9를 I2C1으로 설정했지만, 핀맵 v3.1에서는 TIM4_CH3/CH4 (RB 모터)
- `Motor_ClockAndRemapInit()`에서 I2C1 PE 비활성화 + 클럭 차단 → CLCD/MPU6050은 소프트 I2C 사용 필요
- **위험**: I2C1이 초기화된 뒤 I2C1->CR1 PE를 직접 조작하므로 HAL 상태 불일치 가능

### 5.2 PA3 USART2_RX ↔ NeoPixel 공유

- PA3는 USART2_RX (하드웨어)이면서 동시에 NeoPixel (WS2812) 데이터 출력으로 사용
- `APP_NEOPIXEL_PA3_DYNAMIC_ARBITER_ENABLE`: USART2 유휴 시간에 NeoPixel 업데이트
- USART2 RX가 활성 상태면 NeoPixel 전송을 보류 (2500ms grant)

### 5.3 JSON 상태 5KB 전송

- 매 초 5KB JSON을 USB + BT로 동시 전송
- `HAL_UART_Transmit()` blocking call (timeout 220~420ms)
- 긴 JSON은 BT 미러링 skip (480바이트 초과)

## 7. 개선점 및 문제점

| 문제점 | 영향 | 제안 |
|--------|------|------|
| ❌ **PB8/PB9 I2C1 충돌** | I2C1 초기화 후 레지스터 직접 비활성화 — HAL 상태 불일치 | CubeMX에서 I2C1 제거하고 소프트 I2C만 사용 |
| ❌ **PA3 NeoPixel/USART2 공유** | 동시 동작 불가, 2500ms grant 지연 | 전용 타이머/GPIO로 NeoPixel 분리 |
| ❌ **JSON 5KB blocking TX** | 420ms timeout — 메인 루프 지연 가능 | DMA TX + 링 버퍼 도입 |
| ❌ **SPI LCD 픽셀 단위 전송** | `LCD_DrawChar()`가 픽셀마다 SPI 트랜잭션 | 프레임 버퍼 + DMA SPI 도입 |
| ❌ **ADC 채널 정의 미흡** | 사운드/조이스틱 채널이 하드코딩 | `#define` 또는 config 항목화 |
| ❌ **HAL_UART_Transmit() blocking** | USB + BT 동시 전송 시 2배 지연 | TX DMA + 비동기 큐 |
| ❌ **타이머 Prescaler 수동 설정** | TIM1~4를 코드에서 직접 초기화 (CubeMX 미반영) | .ioc에 TIM1~4 추가 또는 문서화 |
| ❌ **USART3 CubeMX 미포함** | PC10/PC11을 코드에서 GPIO 재설정 | .ioc에 USART3 Partial Remap 추가 |
| ❌ **소프트 I2C 미구현** | I2C1 차단 시 CLCD/MPU6050 경로 없음 | 소프트 I2C bit-bang 라이브러리 추가 |
| ❌ **NeoPixel 정확한 타이밍** | PA3 UART 비트뱅 — 115200 baud → WS2812 타이밍 부정확 | TIM PWM 또는 SPI 전용 채널 사용 |
| ❌ **모터 Init 상태 의존성** | I2C1 비활성화 시점이 CubeMX init 후 | `Motor_Init()` 호출 전까지 I2C1 사용 금지 |
| ❌ **fault 핸들러에서 지연 없음** | `Error_Handler()`가 IWDG 리셋만 믿음 | LED 점멸 또는 상태 저장 후 리셋 |
| ❌ **IR Remote 디코드** | OperatorInputs에서 IRQ 처리 — NEC 프로토콜 리드타임 정확성 | TIM 캡처 또는 외부 디코더 IC |
| ❌ **메모리 사용량** | `s_tx_buf[5200]` + JSON 버퍼 6개 = ~8KB 스택/전역 | 동적 할당 또는 세그먼트 전송 |
| ❌ **부트 지연** | LCD_Init() ~850ms + App_Init ~500ms = 1.3초 | LCD 초기화 최적화 또는 병렬화 |

## **결론**:
* **체계적인 소프트웨어 아키텍처**를 가진 프로젝트. <br>
* 모듈 분리, 설정 집중화, 안전 계층, 비동기 센서, JSON 프로토콜 등 임베디드 시스템 설계의 모범 사례를 다수 적용.
* PB8/PB9 I2C1 충돌과 PA3 NeoPixel 공유가 가장 큰 하드웨어 설계 리스크이며, blocking UART TX가 실시간성의 주된 제약.
* ST7735 LCD + CLCD + NeoPixel + Buzzer를 동시에 구동하는 시각/청각 피드백 시스템은 다른 팀과의 가장 큰 차별점.
