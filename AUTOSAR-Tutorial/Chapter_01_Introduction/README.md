## 2. AUTOSAR란 무엇인가?

### 2.1 정의

**AUTOSAR(AUTomotive Open System ARchitecture)** 는 차량용 전자제어장치(ECU)의 소프트웨어 개발을 위한 **개방형 표준화 아키텍처**입니다.

```
┌─────────────────────────────────────────────────────────────────────┐
│                        AUTOSAR 정의                                 │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  AUT    - AUTomotive     (자동차용)                                 │
│  O      - Open           (개방형)                                   │
│  S      - System         (시스템)                                   │
│  AR     - ARchitecture   (아키텍처)                                 │
│                                                                     │
│  → 자동차용 개방형 시스템 아키텍처                                    │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

### 2.2 핵심 철학

```
╔═══════════════════════════════════════════════════════════════════╗
║                                                                   ║
║   "Cooperate on Standards, Compete on Implementation"             ║
║                                                                   ║
║              표준에서 협력하고, 구현에서 경쟁한다                     ║
║                                                                   ║
╚═══════════════════════════════════════════════════════════════════╝
```

이 철학의 의미:
- **표준 협력**: 모든 회원사가 공통 표준을 함께 개발
- **구현 경쟁**: 표준 기반 위에서 각 회사의 고유 기술로 경쟁

### 2.3 AUTOSAR의 주요 목표

| 목표 | 설명 |
|------|------|
| **재사용성** | 소프트웨어 컴포넌트의 다양한 ECU 및 차량 플랫폼에서 재사용 |
| **교환성** | 서로 다른 공급업체 간 소프트웨어 컴포넌트 교환 가능 |
| **확장성** | 다양한 차량 라인업에 적용 가능한 확장 가능한 솔루션 |
| **하드웨어 독립성** | 특정 하드웨어에 종속되지 않는 소프트웨어 개발 |
| **품질 향상** | 표준화된 개발 프로세스를 통한 품질 향상 |

## 3. AUTOSAR의 역사

### 3.1 탄생 배경 (2000년대 초반)

```
                    2000년대 초반 차량 ECU 개발 현황
                    
┌─────────────────────────────────────────────────────────────────┐
│                                                                 │
│   차량 1대당 ECU 수: 50~100개                                    │
│   소프트웨어 코드량: 1억 라인 이상                                │
│   개발 비용: 전체 차량 개발비의 40% 이상                          │
│                                                                 │
│   문제점:                                                        │
│   ├── ECU별 독자적 소프트웨어 아키텍처                           │
│   ├── 공급업체별 상이한 개발 방식                                │
│   ├── 소프트웨어 재사용 불가                                     │
│   └── 하드웨어 변경 시 전체 소프트웨어 재개발 필요                │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 3.3 버전 체계

AUTOSAR는 두 가지 버전 명명 규칙을 사용합니다:

**Classic Platform (과거)**
```
Release X.Y.Z
예: Release 4.4.0
    │     │ │
    │     │ └── 패치 버전
    │     └──── 마이너 버전
    └────────── 메이저 버전
```

**통합 릴리즈 (현재)**
```
RYY-MM
예: R22-11
    │   │
    │   └── 릴리즈 월 (11월)
    └────── 릴리즈 연도 (2022년)
```

### 6.2 Classic Platform vs Adaptive Platform

| 특성 | Classic Platform | Adaptive Platform |
|------|-----------------|-------------------|
| **대상 시스템** | 실시간, 안전 필수 ECU | 고성능, 유연성 필요 ECU |
| **하드웨어** | 마이크로컨트롤러 (MCU) | 마이크로프로세서 (MPU) |
| **운영체제** | OSEK/VDX 기반 | POSIX 기반 (Linux 등) |
| **아키텍처** | 정적 (컴파일 타임 결정) | 동적 (런타임 결정) |
| **통신** | Signal-based | Service-Oriented (SOA) |
| **업데이트** | 전체 플래싱 | 개별 서비스 업데이트 |
| **적용 사례** | 엔진, 변속기, ABS | ADAS, 인포테인먼트, V2X |
| **개발 언어** | C | C++ (C++11/14) |
| **메모리 관리** | 정적 할당 | 동적 할당 가능 |

### 서술형 문제

**문제 4.** AUTOSAR가 필요하게 된 배경을 기존 ECU 소프트웨어 개발의 문제점과 연관지어 설명하세요.

<details>
<summary>모범 답안</summary>

**모범 답안:**

2000년대 초반, 차량 1대당 50~100개의 ECU가 탑재되면서 소프트웨어 복잡도가 급격히 증가했습니다. 이 시기의 주요 문제점은 다음과 같습니다:

1. **ECU별 독자적 아키텍처**: 각 ECU마다 서로 다른 소프트웨어 구조를 사용하여 통합이 어려웠습니다.

2. **하드웨어 종속성**: 소프트웨어가 특정 하드웨어에 종속되어, MCU 변경 시 전체 소프트웨어를 재개발해야 했습니다.

3. **재사용 불가**: 동일한 기능을 다른 ECU나 차종에 적용하려면 새로 개발해야 했습니다.

4. **공급업체 간 호환성 문제**: 서로 다른 공급업체의 소프트웨어 통합에 많은 시간과 비용이 소요되었습니다.

AUTOSAR는 표준화된 아키텍처와 인터페이스를 제공하여 이러한 문제들을 해결하고, 소프트웨어 재사용성과 개발 효율성을 크게 향상시켰습니다.
</details>

---

**문제 5.** Classic Platform과 Adaptive Platform이 각각 어떤 ECU에 적합한지 예시를 들어 설명하세요.

<details>
<summary>모범 답안</summary>

**모범 답안:**

**Classic Platform 적합 ECU:**
- **엔진 제어 ECU (ECM)**: 실시간 제어가 필수적이며, 결정론적 동작이 요구됩니다.
- **ABS/ESC ECU**: 안전 필수 기능으로, 확정적인 응답 시간이 보장되어야 합니다.
- **변속기 제어 ECU (TCU)**: 하드 실시간 요구사항과 기능 안전이 중요합니다.

**Adaptive Platform 적합 ECU:**
- **ADAS 컨트롤러**: 센서 융합, 머신러닝 기반 인지 등 높은 연산 능력이 필요합니다.
- **인포테인먼트 시스템 (IVI)**: 동적 서비스 로딩, 앱 설치/삭제 등 유연성이 필요합니다.
- **V2X 통신 ECU**: 클라우드 연결, OTA 업데이트 등 서비스 지향 아키텍처가 적합합니다.

선택 기준은 주로 실시간성 요구 수준, 연산 복잡도, 동적 업데이트 필요성에 따라 결정됩니다.
</details>
